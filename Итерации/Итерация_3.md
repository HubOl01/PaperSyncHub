## 1. Бизнес-требования

### 1.1. Цели продукта (с точки зрения владельца/заказчика)

| Идентификатор | Цель | Критерий достижения (метрика) | Приоритет |
|--------------|------|------------------------------|-----------|
| BR-01 | Сокращение времени на вставку ссылок на использованную литературу | Вставка ссылки на источник требует 2 клика | Средний |
| BR-02 | Обеспечение оптимизированного хранения данных | Размер файлов, создаваемых программой, не должен превышать размер файлов в форматах конкурентов более чем на 20%  | Высокий |
| BR-03 | Снижение потерь контекста при переключении между артефактами | Пользователь выполняет 80% рабочих операций в рамках одного интерфейса | Средний |
| BR-04 | Минимизация ошибок версионирования | Автоматическое отслеживание зависимостей между файлами в 95% случаев | Высокий |
| BR-05 | Обеспечение приватности данных (local-first) | Все пользовательские данные по умолчанию хранятся локально, экспорт — только по явному запросу | Высокий |

### 1.2. Ограничения бизнеса
- Бюджет разработки: ограниченный (студенческий проект)
- Сроки: подготовка концепции и ТЗ в рамках текущего семестра
- Ресурсы: команда из 3 человек, частичная занятость
- Лицензирование: использование open-source компонентов там, где это возможно

## 3. Функциональные требования, атрибуты качества, ограничения

### 3.1. Функциональные требования

| Идентификатор | Требование | Описание |
|--------------|------------|----------|
| FR-01 | Управление артефактами | Система позволяет создавать, связывать и редактировать сущности: Статья, Код, Датасет, Заметка, Библиография |

### 3.2. Атрибуты качества

| Категория | Требование | Обоснование |
|-----------|------------|-------------|
| Производительность | Запуск приложения ≤ 3 сек, отклик интерфейса ≤ 200 мс | Для сохранения потока работы исследователя |
| Надёжность | Автосохранение каждые 30 сек, восстановление после сбоев | Предотвращение потери данных при длительной работе |
| Безопасность | Все данные локальны, шифрование экспортируемых архивов по запросу | Соответствие требованиям вузов и приватности |
| Удобство использования | Интерфейс осваивается за ≤ 15 минут, справка встроена | Целевая аудитория — не обязательно технически подготовленные пользователи |
| Расширяемость | Модульная архитектура, плагины для новых языков/форматов | Возможность развития без переписывания ядра |
| Совместимость | Поддержка Windows 10/11, Linux (Ubuntu 20.04+), macOS 12+ | Кроссплатформенность как ключевое требование |

### 3.3. Ограничения

- Техническое: использование AvaloniaUI для кроссплатформенного десктопа
- Ресурсное: минимальные требования к ПК — 4 ГБ ОЗУ, 2 ядра CPU, 10 ГБ свободного места
- Лицензионное: только open-source или бесплатные для образования компоненты
- Юридическое: соблюдение авторских прав при работе с библиографией и внешними данными

### 4.1. Интеграции

| Система | Назначение | Формат обмена | Метод интеграции |
|---------|------------|---------------|------------------|

## 5. MVP (Minimum Viable Product)

### 5.1. Формулировка

PaperSync Hub MVP — это локальное десктоп-приложение для исследователя, которое позволяет:

1. Создать проект и добавить в него 4 типа артефактов: черновик статьи (Markdown+LaTeX), код скрипта (выполняемый в встроенной консоли), датасет (файл или ссылка), библиографическую запись.
2. Связать артефакты между собой и отслеживать зависимости.
3. Выполнить код скрипта прямо в интерфейсе и сохранить результат как часть проекта.
4. Сгенерировать отчёт по упрощённому шаблону ГОСТ (титульник, оглавление, список литературы).
5. Экспортировать весь проект в один ZIP-архив с метаданными.

### 5.2. Критерии готовности MVP

- [ ] Приложение запускается на целевых ОС без ручной настройки
- [ ] Пользователь может создать проект, добавить 4 типа артефактов и связать их
- [ ] Встроенная консоль выполняет простой Python-скрипт и сохраняет вывод
- [ ] Генерация базового PDF-отчёта с титульным листом и списком литературы
- [ ] Экспорт архива, который можно распаковать и открыть на другом компьютере
- [ ] Документация: краткое руководство пользователя (≤ 5 страниц)

### 5.3. Что НЕ входит в MVP

- Полноценная визуализация Git-истории (только базовые коммиты)
- Интеграция с облачными LLM (только локальная модель или заглушка)
- Расширенные шаблоны ГОСТ (только базовая структура)
- Синхронизация между устройствами
- Поддержка всех языков программирования (только Elixir как пилот)
